# Repository Overview

## Purpose

This project is a portable shell-first workstation setup. It is built around four ideas:

- keep user configuration in version control
- install configuration by symlinking into a target home directory
- separate machine- or environment-specific state into `custom/`
- expose recurring personal workflows as executable scripts in `bin/`

## Top-Level Layout

### `install`

Primary installer. It prepares the target home, clones or reuses `custom/`, creates symlinks for dotfiles and XDG config paths, generates environment files, and optionally invokes a custom install hook.

### `demo`

Sandboxed installer for trying the repo without touching the real home directory. It provisions a temporary home at `custom/tmp/demo_home`, installs into it, and opens a shell with the demo environment active.

### `bin/`

The main executable surface. At the time of documentation, the directory contains 106 top-level files and several support directories.

Contents include:

- core dotfiles management commands
- Git helpers
- archive and filesystem utilities
- notifications and automation helpers
- multimedia helpers
- terminal, tmux, and editor wrappers
- scratch helpers and templates
- support directories such as `lib/`, `completion/`, `cronjobs/`, and `deprecated/`

### `config/`

Repository-managed configuration that is linked into the destination home or XDG locations. This includes shell startup files, Vim/Neovim, tmux, ranger, VS Code, i3, Jupyter, feh, cmus, VLC, and more.

### `custom/`

Machine-local or sensitive material that should not live directly in the main repo history. The installer expects this directory to exist and can clone it from a separate repository. It also stores environment-specific overrides, history files, backup state, and custom binaries.

### `notes/`

Tracked as a git submodule according to `.gitmodules`. This appears to be a separate notes repository integrated into the main workspace.

### `resources/`

Static assets used by some scripts or configurations, such as media or reference files.

### `thirdparty/`

External tools and plugins tracked through submodules, including fzf, oh-my-zsh, tmux plugins, dunst, and several others.

### `logs/`

Runtime or archived logs generated by scripts and operational workflows.

### `tmp/`

Ephemeral scratch output.

### `.github/workflows/`

CI workflow that exercises installation on Ubuntu for multiple environments and verifies both `zsh` and `bash` startup.

## Operating Model

The repository behaves more like a small shell platform than a static dotfiles dump:

- `install` defines the installation contract.
- `bin/lib/` contains shared shell primitives.
- `custom-env-resolve` maps a logical config path to the correct environment-specific file.
- `bin/dotfiles` provides a user-facing command dispatcher.
- CI verifies installation and shell startup for multiple environment names.

## Environment Model

The repo is environment-aware. `DOTFILES_ENV` selects a profile, and environment-specific files can live under `custom/env/<env>/...`. When no override exists, the repo falls back to the global `custom/` copy or the tracked file in the main repository.
